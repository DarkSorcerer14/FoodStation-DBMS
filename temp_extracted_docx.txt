ABSTRACT
The Food Delivery System with Multi-Vendor Orders is a database-driven solution designed to manage and streamline online food ordering operations involving multiple restaurants within a single platform. With the rapid growth of digital food delivery services, there is a need for an efficient and structured system capable of handling customers, vendors, food items, orders, payments, and delivery operations in an integrated manner. This project focuses on designing a robust relational database that supports multi-vendor ordering, allowing customers to purchase food items from different restaurants in a single transaction.

The system is developed using the Entity-Relationship (ER) model to identify key entities such as Customer, Vendor, FoodItem, Orders, Payment, and DeliveryPartner. These entities and their relationships are transformed into a normalized relational schema to ensure data consistency, integrity, and minimal redundancy. The OrderItem table plays a crucial role in resolving the many-to-many relationship between Orders and FoodItem, thereby enabling multi-vendor functionality.

The database is implemented using SQL, incorporating Data Definition Language (DDL) commands to create tables with appropriate primary keys, foreign keys, and constraints. Data Manipulation Language (DML) commands are used to insert and manage sample records for testing purposes. Referential integrity and normalization principles are applied to maintain reliability and scalability.

Overall, the system provides a structured foundation for efficient order management, secure payment tracking, and delivery coordination. It demonstrates how a well-designed relational database can effectively support complex business processes in modern online food delivery platforms.

CHAPTER 1
PROBLEM UNDERSTANDING, IDENTIFICATION OF ENTITIES AND RELATIONSHIPS, AND CONSTRUCTION OF DATABASE USING ER MODEL FOR FOOD DELIVERY SYSTEM WITH MULTI-VENDOR ORDERS
1.1 Introduction
The rapid growth of digital platforms has transformed the way people access services, particularly in the food industry. Online food delivery systems have become an integral part of urban and semi-urban lifestyles, offering convenience, accessibility, and variety at the touch of a button. With increasing smartphone penetration and internet connectivity, customers prefer ordering food online rather than visiting restaurants physically. This shift has led to the development of large-scale food delivery platforms that integrate multiple restaurants, customers, delivery partners, and payment systems within a single ecosystem.
A Food Delivery System with Multi-Vendor Orders is a comprehensive database-driven application that enables customers to browse multiple restaurants, select food items from different vendors, place orders, make payments, and track deliveries in real time. Unlike traditional single-vendor ordering systems, a multi-vendor model allows customers to order items from more than one restaurant in a single transaction. This feature introduces additional complexity in managing orders, payments, and delivery coordination.
The backbone of such a system is a well-structured database management system (DBMS). The database stores and manages information about customers, vendors, menu items, orders, payments, and delivery personnel. Without a properly designed database, the system would suffer from inconsistencies, redundancy, and inefficiencies. Therefore, designing an accurate and normalized database schema is essential to ensure data integrity, reliability, and scalability.
This chapter focuses on understanding the problem domain, identifying system requirements, determining the entities and relationships involved, and constructing the database using the Entity-Relationship (ER) model. The ER model provides a conceptual framework that visually represents system components and their interactions before implementation in relational form.
1.2 Motivation
The motivation behind developing the Food Delivery System with Multi-Vendor Orders arises from the increasing demand for efficient and scalable food delivery platforms. Modern consumers expect seamless user experiences, real-time order tracking, secure payment processing, and access to a wide variety of restaurants through a single application.
Traditional food ordering methods, such as phone-based orders or manual billing systems, are inefficient and prone to errors. They lack centralized data management and fail to provide analytical insights. Even early online systems were limited to single-vendor ordering, which restricted customers to purchasing from only one restaurant per transaction.
However, in real-world scenarios, customers may want to order desserts from one restaurant and main courses from another in the same session. Managing such multi-vendor orders requires careful coordination of inventory management, pricing, order splitting, payment allocation, and delivery scheduling. Without a properly structured database, these operations become highly complex and error-prone.
Another motivating factor is the need for data-driven decision-making. Vendors require insights into sales performance, customer preferences, and revenue generation. Delivery partners need efficient order allocation systems to minimize delivery time. Administrators need accurate records for auditing and monitoring platform performance.
Thus, the motivation of this project is to design a robust database system that effectively supports multi-vendor ordering while ensuring data consistency, reduced redundancy, scalability, and enhanced system performance.


1.3 Scope
The scope of the Food Delivery System with Multi-Vendor Orders encompasses all major components required for the smooth functioning of an online food ordering platform. The system is designed to manage data efficiently and support real-time operations.
Customer Management
The system maintains detailed records of customers, including personal information, contact details, and delivery addresses. Customers can register, log in, place orders, and view order history. The database ensures uniqueness of email and phone numbers to avoid duplication.
Vendor Management
Multiple vendors or restaurants can register on the platform. The system stores vendor details such as restaurant name, location, contact number, rating, and operational status. Vendors can update menu items and manage availability.
Food Item Management
Each vendor maintains a list of food items offered. The database stores attributes such as food name, price, category, and availability status. This ensures customers can browse updated menus at any time.
Order Management
Customers can place orders containing multiple food items from different vendors. The system tracks order status, total amount, and timestamps. Multi-vendor support is achieved through an associative entity that connects orders with multiple food items.
Payment Processing
The system records payment details including payment method, transaction date, and payment status. It ensures secure and accurate transaction tracking.

Delivery Management
Delivery partners are assigned to orders. The database tracks pickup time, delivery time, and delivery status to ensure timely service.
The scope is limited to database design and implementation. Front-end interface design and mobile application development are outside the scope of this project.
1.4 Problem Statement
Design and implement a Food Delivery System with Multi-Vendor Orders that efficiently manages customer registrations, vendor listings, food item catalogs, order processing, payment transactions, and delivery tracking. The system must support multi-vendor ordering within a single transaction while maintaining data integrity and consistency.
The database should:
Eliminate redundancy through normalization.
Maintain referential integrity using primary and foreign keys.
Support efficient querying and reporting.
Ensure scalability for increasing users and vendors.
Provide accurate tracking of orders, payments, and deliveries.
1.5 Project Requirements
1.5.1 Functional Requirements
The system shall allow customers to register and log in. The system shall allow vendors to register and manage menus. The system shall enable customers to browse food items. The system shall support multi-vendor orders in a single transaction. The system shall calculate the total order amount automatically. The system shall record payment details securely. The system shall assign delivery partners and track delivery status. The system shall maintain historical data of orders and transactions.
1.5.2 Non-Functional Requirements
Data Integrity: All relationships must enforce referential integrity constraints. Security: Customer and payment information must be securely stored. Scalability: The system must handle increasing transaction volumes. Performance: Queries must retrieve data efficiently without delay. Reliability: The system should prevent data inconsistency and orphan records.
1.6 Identification of Entity and Relationships
To design the database effectively, it is essential to identify all entities involved in the system and understand their relationships.
Entities Identified
Customer
Attributes: CustomerID, Name, Email, Phone, Address, RegistrationDate
Each customer can place multiple orders.
Vendor
Attributes: VendorID, VendorName, Location, ContactNumber, Rating, Status
Each vendor can offer multiple food items.
FoodItem
Attributes: FoodID, FoodName, Price, Category, AvailabilityStatus, VendorID
Each food item belongs to one vendor.
OrderItem
Attributes: OrderItemID, Quantity, PriceAtOrderTime, OrderID, FoodID, VendorID
Resolves many-to-many relationship between Orders and FoodItem.
Payment
Attributes: PaymentID, PaymentMethod, PaymentStatus, TransactionDate, OrderID
Each order has one payment record.

DeliveryPartner
Attributes: DeliveryPartnerID, Name, Phone, VehicleType, AvailabilityStatus
Each delivery partner can handle multiple deliveries.


Relationships Identified
A Customer places Orders (1:N relationship).
A Vendor provides FoodItems (1:N relationship).
An Order contains multiple FoodItems, and a FoodItem appears in multiple Orders (M:N relationship resolved using OrderItem).
An Order has one Payment (1:1 relationship).
An Order is assigned to one Delivery (1:1 relationship).
A DeliveryPartner handles multiple Deliveries (1:N relationship).

CHAPTER 2
DESIGN OF RELATIONAL SCHEMA, CREATION OF DATABASE AND TABLES FOR FOOD DELIVERY SYSTEM WITH MULTI-VENDOR ORDERS
2.1 Relational Schema for Food Delivery System with Multi-Vendor Orders
The relational schema represents the logical database structure derived from the conceptual Entity-Relationship (ER) model discussed in Chapter 1. While the ER diagram provides a high-level abstraction of entities and their interrelationships, the relational schema translates this conceptual model into a structured form suitable for implementation using a Relational Database Management System (RDBMS).
In the Food Delivery System with Multi-Vendor Orders, the relational schema is designed to support efficient storage, retrieval, and manipulation of data related to customers, vendors, food items, orders, payments, and delivery operations. The schema ensures elimination of redundancy through normalization and maintains data integrity using primary keys, foreign keys, and appropriate constraints.
The system consists of the following relations:
Customer (CustomerID, Name, Email, Phone, Address, RegistrationDate)
Vendor (VendorID, VendorName, Location, ContactNumber, Rating, Status)
FoodItem (FoodID, FoodName, Price, Category, AvailabilityStatus, VendorID)\
Orders (OrderID, OrderDate, TotalAmount, OrderStatus, CustomerID)
Payment (PaymentID, PaymentMethod, PaymentStatus, TransactionDate, OrderID)
DeliveryPartner (DeliveryPartnerID, Name, Phone, VehicleType, AvailabilityStatus)
The OrderItem table plays a critical role in supporting multi-vendor functionality. Since an order may contain multiple food items from different vendors, the many-to-many relationship between Orders and FoodItem is resolved using this associative table.
The schema has been normalized up to Third Normal Form (3NF) to remove transitive dependencies and ensure minimal redundancy. Each table contains attributes that are fully functionally dependent on the primary key. Referential integrity is enforced through foreign key constraints, ensuring consistency across related tables.

2.2 Description of Tables
This section provides a comprehensive explanation of each table in the system, including its purpose, attributes, and constraints.

Table 2.1: Customer
Purpose: The Customer table stores information about registered users who place food orders through the platform. Each customer is uniquely identified and can place multiple orders.
Primary Key: Customer_ID (INTEGER)
Data Types and Constraints:
CustomerID: INTEGER, Primary Key, Not Null, Auto Increment
Name: VARCHAR(100), Not Null
Email: VARCHAR(100), Not Null, Unique
Phone: VARCHAR(15), Not Null, Unique
Address: TEXT, Not Null
The unique constraints on Email and Phone ensure that duplicate customer accounts cannot be created. This improves data integrity and prevents redundancy.
Table 2.2: Vendor
Purpose: The Vendor table maintains information about restaurants registered on the platform. Vendors can upload menus and receive orders.
Primary Key: VendorID (INTEGER)
Data Types and Constraints:
VendorID: INTEGER, Primary Key, Not Null, Auto Increment
VendorName: VARCHAR(100), Not Null
Location: TEXT, Not Null
ContactNumber: VARCHAR(15), Unique
Rating: DECIMAL(2,1), Check (Rating BETWEEN 1 AND 5)
The rating attribute allows tracking vendor performance, while the status attribute indicates whether the vendor is active or inactive.
Table 2.3: FoodItem
Purpose: Stores detailed information about menu items offered by vendors.
Primary Key: FoodID (INTEGER)
Foreign Key: VendorID references Vendor(VendorID)
Data Types and Constraints:
FoodID: INTEGER, Primary Key, Not Null, Auto Increment
FoodName: VARCHAR(100), Not Null
Price: DECIMAL(10,2), Check (Price > 0)
Availability: VARCHAR(20), Not Null
VendorID: INTEGER, Foreign Key
Each food item is linked to a vendor, establishing a one-to-many relationship between Vendor and FoodItem.

Table 2.4: Orders
Purpose: Records details of orders placed by customers.
Primary Key: OrderID (INTEGER)
Foreign Key: CustomerID references Customer(CustomerID), Food_ID references Food(Food_ID)
Data Types and Constraints:
OrderID: INTEGER, Primary Key, Not Null, Auto Increment
OrderDate: DATETIME, Not Null
TotalAmount: DECIMAL(10,2), Check (TotalAmount >= 0)
OrderStatus: VARCHAR(20), Not Null
CustomerID: INTEGER, Foreign Key
An order belongs to exactly one customer, but a customer can place multiple orders.

Table 2.5: Payment
Purpose: Stores transaction details related to order payments.
Primary Key: PaymentID (INTEGER)
Foreign Key: OrderID references Orders(OrderID)
Data Types and Constraints:
PaymentID: INTEGER, Primary Key, Not Null, Auto Increment
PaymentMethod: VARCHAR(50), Not Null
PaymentStatus: VARCHAR(20), Not Null
Date: DATETIME, Not Null
OrderID: INTEGER, Foreign Key
Each order is associated with one payment record.
Table 2.6: DeliveryPartner
Purpose: Maintains records of delivery personnel.
Primary Key: DeliveryPartnerID (INTEGER)
Data Types and Constraints:
DeliveryPartnerID: INTEGER, Primary Key, Not Null, Auto Increment
Name: VARCHAR(100), Not Null
Phone: VARCHAR(15), Unique
VehicleType: VARCHAR(50)
TimeTaken: VARCHAR(20), Not Null

2.3 Creation of Database and Tables – DDL Commands
CREATE DATABASE FoodDeliveryDB;
USE FoodDeliveryDB;

CREATE TABLE Customer (
    CustomerID INT PRIMARY KEY,
    Name VARCHAR(100) NOT NULL,
    Email VARCHAR(100) UNIQUE NOT NULL,
    Phone VARCHAR(15) UNIQUE NOT NULL,
    Address TEXT NOT NULL
);

CREATE TABLE Vendor (
    VendorID INT PRIMARY KEY,
    VendorName VARCHAR(100) NOT NULL,
    Location TEXT NOT NULL,
    ContactNumber VARCHAR(15) UNIQUE,
    Rating DECIMAL(2,1) CHECK (Rating BETWEEN 1 AND 5)
);

CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    OrderDate DATETIME NOT NULL,
    TotalAmount DECIMAL(10,2),
    OrderStatus VARCHAR(20) NOT NULL,
    CustomerID INT,
    FOREIGN KEY (CustomerID) REFERENCES Customer(CustomerID)
        ON UPDATE CASCADE
        ON DELETE CASCADE
);

CREATE TABLE FoodItem (
    FoodID INT PRIMARY KEY,
    FoodName VARCHAR(100) NOT NULL,
    Price DECIMAL(10,2) CHECK (Price > 0),
    AvailabilityStatus VARCHAR(20) NOT NULL,
    VendorID INT,
    FOREIGN KEY (VendorID) REFERENCES Vendor(VendorID)
        ON UPDATE CASCADE
        ON DELETE CASCADE
);

CREATE TABLE Payment (
    PaymentID INT PRIMARY KEY,
    PaymentMethod VARCHAR(50) NOT NULL,
    PaymentStatus VARCHAR(20) NOT NULL,
    Date DATETIME NOT NULL,
    OrderID INT,
    FOREIGN KEY (OrderID) REFERENCES Orders(OrderID)
        ON UPDATE CASCADE
        ON DELETE CASCADE
);

CREATE TABLE DeliveryPartner (
    DeliveryPartnerID INT PRIMARY KEY,
    Name VARCHAR(100) NOT NULL,
    Phone VARCHAR(15) UNIQUE,
    VehicleType VARCHAR(50),
    TimeTaken VARCHAR(20) NOT NULL
);

CREATE TABLE PaymentLog (
    LogID INT AUTO_INCREMENT PRIMARY KEY,
    OrderID INT,
    PaymentID INT,
    Status VARCHAR(20),
    LogTime DATETIME
);

CREATE TABLE CustomerStats (
    CustomerID INT PRIMARY KEY,
    TotalOrders INT DEFAULT 0,
    TotalSpent DECIMAL(10,2) DEFAULT 0,
    AvgSpent DECIMAL(10,2) DEFAULT 0
);

)

    INSERT INTO Orders (OrderDate, TotalAmount, OrderStatus, CustomerID)

    SELECT 'Order inserted successfully' AS Message;
END$$

        INSERT INTO PaymentLog (OrderID, PaymentID, Status, LogTime)

CREATE TABLE CustomerTotal (
    CustomerID INT PRIMARY KEY,
    TotalSpent DECIMAL(10,2)
);

DELIMITER $$

CREATE PROCEDURE calc_customer_total()
BEGIN
    DECLARE done INT DEFAULT FALSE;
    DECLARE cid INT;
    DECLARE amt DECIMAL(10,2);

    DECLARE cur CURSOR FOR
        SELECT CustomerID, TotalAmount FROM Orders;

    DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

    OPEN cur;

    read_loop: LOOP
        FETCH cur INTO cid, amt;
        IF done THEN
            LEAVE read_loop;
        END IF;

        INSERT INTO CustomerTotal (CustomerID, TotalSpent)
        VALUES (cid, amt)
        ON DUPLICATE KEY UPDATE
            TotalSpent = TotalSpent + amt;

    END LOOP;

    CLOSE cur;
END$$

DELIMITER ;

ANALYZING THE PITFALLS, IDENTIFYING THE DEPENDENCIES, AND APPLYING NORMALIZATIONS
 
4.1  Analyse the Pitfalls in Relations
Before normalization is applied, the raw data from the Food Delivery Management System was stored in a single flat, denormalized table. This approach introduces several serious problems:
 
Pitfall 1 — Data Redundancy
Customer details (Name, Email, Phone, Address) are repeated for every order they place. Vendor names are duplicated for every food item ordered from that vendor. This wastes storage and increases the chance of inconsistencies.
 
Pitfall 2 — Update Anomaly
If a customer changes their phone number, every row containing that customer must be updated. Missing even one row causes inconsistent data across the database.
 
Pitfall 3 — Insertion Anomaly
A new food item cannot be recorded without associating it with an existing order. This forces artificial entries just to store legitimate vendor/food data.
 
Pitfall 4 — Deletion Anomaly
Deleting the only order of a customer results in losing that customer's contact information entirely.
 
Pitfall 5 — Multi-valued Columns (Non-atomic)
The 'FoodItems' column stores multiple items in a single cell (e.g., 'Butter Chicken, Garlic Naan'), violating atomicity and making queries on individual items impossible.
 
Pitfall 6 — Derived / Redundant Columns
The 'AvgSpent' column in CustomerStats is mathematically derivable from TotalSpent / TotalOrders. Storing it redundantly risks inconsistency if the base values are updated without recalculating it.

UNNORMALIZED TABLE


 

4.2  First Normal Form (1NF)
4.2.1  Identify Dependency
A relation is in 1NF if every attribute contains only atomic (indivisible) values and there are no repeating groups or multi-valued columns. 

In the flat table, the column FoodItems contains comma-separated lists such as:
'Butter Chicken, Garlic Naan'   →   Non-atomic. Violates 1NF. 

This prevents querying individual food items, applying price computations, or joining with the FoodItem table. Additionally, all columns must have a well-defined primary key.

1NF Violation: FoodItems column stores multiple values in one cell → must be decomposed into one row per food item.

5.1 Introduction to Transactions
A transaction is a sequence of database operations that are executed as a single logical unit of work. In the Food Delivery System, transactions ensure that operations such as placing an order, processing payment, and assigning delivery are executed reliably and consistently.
A transaction must satisfy ACID properties to maintain database integrity.
5.1.1 Properties
Atomicity: Ensures that all operations in a transaction are completed successfully. If any operation fails, the entire transaction is rolled back.
Consistency: Ensures that the database moves from one valid state to another, maintaining all constraints.
Isolation: Ensures that concurrent transactions do not interfere with each other.
Durability: Once a transaction is committed, the changes are permanently stored in the database.
5.1.2 States
A transaction passes through the following states:
Active – Transaction is being executed
Partially Committed – Final statement executed
Committed – Changes are permanently saved
Failed – Transaction cannot proceed
Aborted – Changes are rolled back 
5.2 Transaction Control Language (TCL)
Transaction Control Language commands are used to manage transactions in the database.
5.2.1 Save point
SAVEPOINT is used to create a point within a transaction to which we can later roll back.
5.2.2 Commit
COMMIT permanently saves all changes made during the transaction.
5.2.3 Rollback
ROLLBACK is used to undo changes made in a transaction or to revert to a specific savepoint.

6.3.2 Order Management APIs (/api/orders)
The order endpoints handle the complete lifecycle of a customer's purchase. A POST request to /api/orders inserts a new order record into the database, capturing the customer ID, vendor ID, food items, and total amount. A PUT request to /api/orders/:id/status allows vendors to progress an order through its status stages: Placed → Preparing → Out for Delivery → Delivered.

javascript
// app_server.js 
app.post('/api/orders', (req, res) => {
  const { total_amount, customer_id } = req.body;
  const status = 'Placed';

  db.query('SELECT MAX(OrderID) as maxId FROM orders', (err, rows) => {
    if (err) return res.status(500).json({ error: err.message });
    
    const nextOrderId = (rows[0].maxId || 0) + 1;
    const query = 'INSERT INTO orders (OrderID, OrderDate, TotalAmount, OrderStatus, CustomerID) VALUES (?, NOW(), ?, ?, ?)';
    
    db.query(query, [nextOrderId, total_amount, status, customer_id || 1], (insertErr) => {
      if (insertErr) return res.status(500).json({ error: insertErr.message });
      res.json({ success: true, orderId: nextOrderId });
    });
  });
});

app.put('/api/orders/:id/status', (req, res) => {
  const { status } = req.body;
  const { id } = req.params;
  db.query('UPDATE orders SET OrderStatus = ? WHERE OrderID = ?', [status, id], (err) => {
    if (err) return res.status(500).json({ error: err.message });
    res.json({ success: true });
  });
});

7.7 Discussion
The integration of a seeded relational MySQL database with a lightweight Node.js and Express.js backend demonstrates the practical effectiveness of applying database fundamentals to a real-world multi-vendor e-commerce scenario. By centralizing all data persistence within a structured MySQL schema and exposing it through a clean REST API, the system achieves a clear and maintainable separation between the data tier and the presentation tier.
The decision to use vanilla HTML, CSS, and JavaScript for the frontend eliminated build-tool complexity while still enabling fully dynamic, API-driven interfaces across all three role-based portals. This approach proved particularly effective for a DBMS-focused academic project, keeping the frontend lightweight and ensuring that database operations remain the architectural centerpiece.
The multi-vendor checkout flow wherein a single order aggregates items from multiple restaurants within one transaction directly exercised the relational integrity of the schema, validating the foreign key relationships between the orders, customers, vendors, and food_items tables under realistic usage conditions.
7.8 Conclusion
The FoodStation Multi-Vendor Food Delivery System successfully achieves its core objective: providing a functional, role-segregated platform for managing food ordering operations across multiple vendors within a unified database environment.
The project bridges fundamental DBMS concepts including relational schema design, referential integrity, transactional INSERT and UPDATE operations, and seed-based data initialization with practical full-stack web development using Node.js and Express.js. The resulting system is operationally correct at the data tier, functionally complete at the logic tier, and fully accessible through dedicated browser-rendered interfaces for each user role, proving its overall success as a comprehensive database management simulation.
